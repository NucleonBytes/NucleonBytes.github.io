Release Notes

API v2.0 Out Now!

API v2.0 is now out with many new features and improvements. This API version does not affect legacy API code. This release brings:


  • New fs.fileSync method returns a list of files in a given directory
  • New fs.browseDir method opens folder dialog and returns selected path
  • the app.userinput method replaces the inputbox
  • New fs.checkFolder returns boolean value for folder existance
  • The system.path method has been updated for a much cleaner and platform independant value
  • system.run now takes in two value (path and argument) to run a program and returns a processID
  • New system.killProcess kills processes on the machine by process ID

Basics

Getting started!

This documentation has been written to help developers get into grips with the whole concept and approach to web-application development in NitroJS. Our API documentation is meant to provide detailed (yet easy to understand) information about a given feature or method.


By using this documentation, you should be able to learn and utilise NitroJS and its components. Please remember, this documentation isnt just for those starting out in web-application development, it is also for those legendary developers out there looking for a reference.


We also have an option for those who learn better while working; Head over to our kickstart New! page, to get started with our mini step-by-step NitroJS development guide!


Version

As we continue to develop and expand the capabilities of NitroJS and its components, we may have to break older API in order to ensure NitroJS remains a secure platform. This can be annoying if you have a large code base, therefore, we have created an API versioning system which allows you to force the old API even after a modular API change. We do NOT recommend this as it may render your application unsafe in the future.


The default API-blocks (aka block modules) will not be changed. We will only ever add to the Block modules list.


Manifest Updated!

Every NitroJS application has to start from a manifest file called the json-nitro-10 (JNX). Manifests in NitroJS are written with the JSON syntax (javascript object notation). They provide the NitrixCore with information about the web-app it is about to run. Manifest files contains things like name, version, runtime information, icons etc. Web-streamed application also require a manifest file. All NitroJS application manifests must have the '.jnx' extension.



	{
		"manifest_version":3,
		"application":{
			"name":"my Application",
			"version": "2.0.1.6",
			"homepage":"*/index.html",
			"icon":"*/star.ico",
			"permissions":["fs","app"]
		},
		"runtime":{
			"version":"1.0.0.6",
			"gpu_acceleration":false,
			"api":"v1.2"
		}
	}

Asterisks are used to get the current directory of the manifest. Manifest version is required in order for NitroJS to work correctly. Here is what each version allows:


Version Action
1 Retreives Application Name, Version and Homepage from the manifest
2 Retreives Application Name, Version, Homepage and Icon path from the manifest
3 Retreives Application Name, Version, Homepage, Icon path, gpu acceleration and Runtime Version from the manifest

Permissions

In order for your application to access system level information and capabilities, you will need to declare something called permissions. Permissions give you extended features and access to parts of the system, enabling you to utilize the whole environment. In order to run block Modules, you will need to request permission to use them from NitroJS, in turn preventing unauthorised access. Although permissions is a block module, it does not have to be declared itself. Setting permission is easy, permissions are defined in the manifest under the application object properties:



	{
		"manifest_version":3,
		"application":{
			"name":"my Application",
			"version": "2.0.1.6",
			"homepage":"*/index.html",
			"icon":"*/star.ico",
			"permissions":["fs","app"]
		},
		"runtime":{
			"version":"1.0.0.6",
			"gpu_acceleration":false,
			"api":"v1.2"
		}
	}

To remove permissions:



	permission.remove("fs"); //access to fs will now be denied
	permission.remove("storage"); //access to storage now denied

Hello World

Getting started with NitroJS just as easy as making a basic HTML webpage. However, if you want to add awesome functionalities to your application, you will need to set and initialise NitrixCore. This is an example of how you would set things up:



	var Nitro = new NitroCore(); //sets new object to be used
	Nitro.import("jquery"); //prepares library to be imported when needed
	
	var appName = app.get("applicationName"); //gets the application name
	var username = app.get("currentUser"); //gets name of user currently logged on
	
	alert("Hello " + username + ", you are currently using " + appName);

Advanced

NitrixCore

The NitrixCore is the main engine behind all of the content computation. It works together with NitroJS and the NitroCore to provide user-level access and information, as well as entry-level communication with the system. In other words, the NitrixCore provides native-feel and speed while the NitroCore and NitroJS prepares and formats the web documents to be processed by the NitrixCore :)


Web-Streaming

The NitrixCore (with the help of the NitroJS) gives you the ability to stream your web-application directly to your client-side. This allows you to push updates almost instantly; as soon as an update is made, your users will automatically be running the latest version all without having to restart the NitroJS environment. Note: The downside to this is that your users will be prompted to grant access to the permissions you define every-time your app runs.



Applciation Options

app.setOpt

These features are provided by the app domain. They allow you to set application-root-level options. For example, you can remove the border of your application, creating an immersive experience for your user, or use system theme to unify elements and components in your webapp. 



	app.setOpt("system.style",true);

The example above will force Nitro to use dsys, the default Nitro-System style.



    app.setOpt("system.immersive", true);

The sample above will load your NitroJS webapp in immersive mode, removing borders and control-bar buttons and titles.


This option should be applied as soon as your application is loaded. Once immersive mode is active, you should NOT attempt to deactivate it as it might break your webapp. Once you activate this option, you will have to handle the closing, resizing and positioning of your application yourself!

debug.setOpt

This feature is provided by the debug domain. It allows you to set application-root-level options. 


    debug.setOpt("notification.error",true);

The above is provided by the debug domain. By setting the notification.error byte to true, you will force NitroJS to inform you whenever an error occurs in your application.


This is useful for when you are debugging your app.

Block Module

app

app.get

The app domain provides access to the application and its direct-entities. Using this domain, you can access methods bound to the application, fetch information about the application and its interactions as well as communicate with the environment running your web-app. One of the methods provided by the app domain is the get method; it allows you to fetch information about the application and its interactions. 



	app.get('applicationName'); //returns the application name
	app.get('applicationVersion'); //returns the application version
	app.get('runtimeVersion'); //returns the runtime version
	app.get('Name'); //returns the text displayed in the titlebar
	app.get('currentUser'); //returns the name of user currently logged in
	app.all(); //returns an object of all the information as properties

app.close

This method is provided by the app domain and allows you to close the environment handling your application


    app.close(); //asynchronously closes the application

The close() method is asynchronous.

The close() method will throw an error if the 'preventClose' byte is set to true

app.maximize

This method is provided by the app domain and allows you to maximize the environment handling your application


    app.maximize();

app.minimize

This method is provided by the app domain and allows you to minimize the environment handling your application


    app.minimize();

app.normal

This method is provided by the app domain and allows you to restore the environment handling your application from a maximized or minimized state.


    app.normal();

app.restart

This method is provided by the app domain and allows you to restart the environment handling your application


    app.restart(); //restarts the runtime

Unlike the close() method, the restart() method isnt async.

The restart() method should not be called manually, (run a new instance with parameters then call close instead)

app.{windowState}

These methods are provided by the app domain. They allow you to set the windows state of the environment handling your application


    app.maximize();
    app.minimize();
    app.normal();

app.showBorder

This method is provided by the app domain. Use this method to set the visibility of your webapp's borders


    app.showBorder(true); //Makes the border visible
    app.showBorder(false); //Makes the border invisible

app.setPos

This method is provided by the app domain. NitroJS provides you with the ability to set the location of the webapp in relation to its parent (in most cases, this is the desktop). 


    app.setPos(12,300); //Sets the x location to 12pixels and the y location to 300pixels

app.setSize

This method is provided by the app domain. This method allows you to resize the environment/window of your webapp


    app.setSize(850,600); //Sets the width to 850pixels and height to 600pixels

app.getPos

This method is provided by the app domain. It returns a location object with x and y values


    var location = app.getPos();
    return location.x; //or location["x"]
    return location.y; //or location["y"]

app.getCursor

This method is provided by the app domain. It returns a location object with x and y values. local values return position of the mouse within the client while the other values return position of mouse within the screen


    var location = app.getCursor();
    return location.localX; //or location["localX"]
    return location.localY; //or location["localY"]
    return location.x; //or location["x"]
    return location.y; //or location["y"]

app.getSize

This method is provided by the app domain. This method works the same as getPos as it returns a size object with width and height values


    var mysize = app.getSize();
    return mysize.width; //or mysize["width"]
    return mysize.height; //or mysize["height"]

app.setMaxSize

This method is provided by the app domain. It allows you to set the maximum size for your webapp's container


    app.setMaxSize("height",500); //Sets the maximum height for the container to 500pixels

app.userinput

This method is provided by the app domain. This method replaces the inputbox() function with deep integration with the NitroJS 'freeze' feature


    var age = app.userinput();
    return age; //returns the value entered by the user

permission

permission.remove

This method is provided by the permission domain. It allows you to remove permissions that have been defined in the application manifest. This is useful for when you are updating the user's license to use your webapp (if applicable). 


    permission.remove("fs"); //removes access to the filesystem

To re-add permissions, you will need to restart the webapp

permission.list

This method is provided by the permission domain. Calling this method will yeild a read-only list populated with permissions currently active in the webapp. 


    permission.list();

system

system.run

This method is provided by the systems domain. In order to run third-party applications, you will need to call the run method from the system domain. This method takes in two values; the application path (of the application you want to run) and any parameters you want to use as a string. 


    system.run("NitroJS.exe","/application/calc.jnx"); //will execute the NitroJS runtime and make it run the calculator webapp.

As of API v2.0, the security engine in NitroJS prevents you from using the run() method to run an application with administrative rights if NitroJS is not running with administrative rights.

system.killProcess

This method is provided by the systems domain. This method takes in the process ID of the process you would like to kill as a string. 


    system.killProcess(note_process_id);

If you are not running with root or admin rights, killProcess will only be able to end processes launched by NitroJS and processes launched with user rights.

system.path

This method is provided by the systems domain. Finding the path to the commonly used directories can become quite a hassle, therefore, NitroJS provides you with the ability to quickly get the paths of common folders such as 'My documents' and 'Desktop'. 


        system.path("appdata");
	system.path("mydocuments"); 
	system.path("mycomputer");
	system.path("desktop");
	system.path("mymusic");
	system.path("mypictures");
	system.path("myvideos");
	system.path("root"); //returns the directory where the operating system is installed
	system.path("runtime-path"); //returns the folder where NitroJS is installed

system.execute

This method is provided by the systems domain. Some applications require the CMD prompt or Terminal in order to work correctly. Using this method, script files are executed via the terminal which stays active until the script has exited. 


    system.execute("quickClean.bat"); //Will run the batch file via Terminal or CMD prompt

We suggest that you use the run() method instead as it is cleaner and provides more control.

networking

net.secureDownloadFile

This method is provided by the networks domain. It provides you with the means to download files using login credentials for that host. 


    var creds = {username:"root", password:"melon123"};
    var localpath = system.path("appdata");
    net.secureDownloadFile("http://example.com/file.txt", localpath, creds.username, creds.password, DownloadCompleted(), progressChange());
    //Using callback functions for progress change and completion

Callbacks are not required but can be helpful. If you are using callbacks, you must use both callbacks (i.e. You cannot use only one callback)

net.downloadFile

This method is provided by the networks domain. It provides you with the means to download files. 


    var localpath = system.path("appdata");
    net.downloadFile("http://example.com/file.txt", localpath, DownloadCompleted(), progressChange());
    //Using callback functions for progress change and completion

Callbacks are not required but can be helpful. If you are using callbacks, you must use both callbacks (i.e. You cannot use only one callback)

net.uploadFile

This method is provided by the networks domain. It provides you with the means to upload files using login credentials (or not) for that host. 


    var myAppdata = system.path("appdata");
    net.uploadFile(myAppdata+"/file.txt", "http://example.com/file1.txt", "username", "password");
    //Using credentials
    net.uploadFile(myAppdata+"/file.txt", "http://example.com/file1.txt");
    //Without credentials

No callbacks can be used for uploading

net.uploadFileFTP

This method is provided by the networks domain. It provides you with the means to upload files using login credentials (or not) by FTP (file transfer protocols). 


    var myAppdata = system.path("appdata");
    net.uploadFileFTP(myAppdata+"/file.txt", "http://example.com/file1.txt", "username", "password");
    //Using credentials
    net.uploadFileFTP(myAppdata+"/file.txt", "http://example.com/file1.txt");
    //Without credentials

No callbacks can be used for uploading

net.ping

This method is provided by the networks domain. It pings the website and returns availability as boolean 


    net.ping("http://google.com");

net.available

This method is provided by the networks domain. It checks if you are connected to the internet and returns boolean. 


    net.available();

filesystem

fs.createFile

Use the following to create a file. 


    var filepath = system.path("desktop") +"/file.txt"; 
    fs.createFile(filepath)

fs.writeFile

Use the following to write to a file. The following example writes the word "world" to the file. 


    var filepath = system.path("desktop") +"/file.txt"; 
    fs.writeFile(filePath,'world',true); //the last parameter set to true to append to the file

fs.readFile

Use the following to read a file. The following reads the file and returns the content. 


    var filepath = system.path("desktop") +"/file.txt"; 
    var content = fs.readFile(filepath); 
    alert(content);

fs.deleteFile

Use to safely delete a file. 


    var filepath = system.path("desktop") +"/file.txt"; 
    fs.deleteFile(filepath);

fs.checkFile

Use to check if a file exists. 


    var filepath = system.path("desktop") +"/file.txt"; 
    if (fs.checkFile(filepath) == true) {
        alert("file exists");
    }
    else
    {
        alert("file missing");
    }

fs.moveFile

To move a file from one directory to another. 


    var filepath = system.path("desktop") +"/file.txt";
    var newpath = system.path("mydocuments") + "/other-folder/file.txt" 
    fs.moveFile(filePath, newpath);

fs.deleteFile

Use to safely delete a file. 


    var filepath = system.path("desktop") +"/file.txt"; 
    fs.renameFile(filepath,"anotherName.txt");

fs.moveFolder

To move the folders to another location. 


    fs.moveFolder(oldFolderLocation,newLocation);

The 'oldFolderLocation' must be the path and name of the folder, while the 'newLocation' is only the path of the new location.

fs.renameFolder

To move change the name of a folder. 


    fs.renameFolder(oldFolderNameAndPath,newName);

fs.createFolder

To create a new folder, use the following: 


    fs.createFolder(folderNameAndPath);

fs.fileSync

This feature is provided by the FS module. It allows you to get a list of files from a specified folder path. 


    fs.fileSync(folderPath); //returns a list of strings

This function is not asynchronous, therefore it may take time before the list is populated.

fs.browseDir

This feature is provided by the FS module. It allows the user to select a folder, and returns a path. 


    fs.browseDir(); //returns selected path to folder

fs.checkFolder

This feature is provided by the FS module. It allows you to check whether a specified folder exists. 


    fs.checkFolder(); //returns boolean value

storage

storage.store

You can use this module to temporarily store data. Any data you store in Nitro will only exist as long as the application is running. 


    storage.store("myname", "bob"); 
    // storing the name 'bob' with a key called 'myname'

Keys are used to retrieve your value from storage. Avoid using the same key for multiple values as Nitro will return last value

storage.fetch

You can use this module to temporarily store data. Any data you store in Nitro will only exist as long as the application is running. 


    var name = storage.fetch("myname"); 
    // retreives value stored with the key 'myname'

debug

debug.getErrorList

Whenever Nitro runs into a manageable error, it reports it to NitroJS which keeps a record of it. Using this module, you can retrieve the list of all the errors recorded. 


    debug.getErrorList();
    //returns a string-list of every error

debug.logError

Whenever Nitro runs into a manageable error, it reports it to NitroJS which keeps a record of it. Using this module, you can add your own errors to the error list. 


    debug.logError("Error Title", "Error Message");

Components
Events

startup

This event is triggered by NitroJS whenever your application's view starts up. This event is called before the load event. To add a script to this event's function, simply create a function with this name to be executed when the event is triggered. 


    //example
    function startup(){
        //code that you want to run goes here
    }

load

Just like startup, the load event is triggered by NitroJS whenever the application loads. This event is called before the draw event. To add statements scripts to this event, simply create a function with the event name and NitroJS will automatically override the default functions. 


    //example
    function load(){
        //code that you want to run goes here
    }

draw

The draw event is called by NitroJS whenever the application is drawn on screen (i.e. When page changes and when screen is updated). 


    //example
    function draw(){
        //code that you want to run goes here
    }

The draw event gets called very often. To avoid lag, minimize the number of execution to be made on event call

shutdown

This event is called whenever NitroJS is about to shutdown. 


    //example
    function shutdown(){
        //code that you want to run before Nitro shuts down goes here
    }

This event is not called in all circumstances (i.e. when an error or premature shutdown occurs)

renderProcessCreated

This event is called whenever NitroJS creates another process renderer (usually as a result of heavy drawing or high resolution and graphics smoothing). 


    //example
    function renderProcessCreated(){
        //code that you want to run goes here
    }

This event is not called in all circumstances.
